# Phase 08 — Tutor Matching

## Goal
Let students submit in-app tutor requests. No more mailto: links — students fill a quick form, pick their subject + preferred days, add an optional message, and submit. Their pending/accepted requests appear in a "My Requests" section at the top of the Tutors tab.

## DB Schema

```sql
tutor_requests (
  id             uuid PK default gen_random_uuid()
  student_id     uuid REFERENCES profiles(id) ON DELETE CASCADE
  tutor_id       uuid REFERENCES tutors(id) ON DELETE CASCADE
  subject        text NOT NULL
  message        text
  preferred_days text[] default '{}'
  status         text CHECK IN ('pending', 'accepted', 'declined') DEFAULT 'pending'
  created_at     timestamptz default now()
)
```

RLS:
- Authenticated students can SELECT only their own requests (`student_id = auth.uid()`)
- Authenticated students can INSERT only their own requests (`student_id = auth.uid()`)
- No client UPDATE/DELETE (admin-only via Supabase dashboard)

- `getMyTutorRequests()` — fetches student's own requests, joins tutor name

## UI Features

### TutorCard changes
- "Contact" (mailto:) button replaced with "Request" button
- If student has a pending request with this tutor: button shows "Pending ⏱" (disabled)
- Alert shown if student taps a tutor they already requested

### TutorRequestModal (bottom sheet)
- Slides up from bottom with dim backdrop
- Subject selector: pill list from tutor's subjects (first pre-selected)
- Preferred Days: multi-toggle from tutor's availability (optional)
- Message: multi-line text input (optional)
- "Send Request" → saves to DB, closes modal, refreshes list
- "Cancel" or tap backdrop to dismiss

### My Requests section
- Horizontal scroll of compact cards (130px wide)
- Shows pending + accepted requests (declined hidden)
- Each card: tutor name, subject, status badge (Pending/Accepted)
- Appears above the tutor list as FlatList ListHeaderComponent
- Refreshes automatically after every request submission
